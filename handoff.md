## Known Gaps (explicit, not silently missing)
- Windows agent: security scanning and remediation are real no-ops (log a placeholder message, don't fake findings); only the core telemetry/WS loop, crypto-miner detection, and Application Monitoring's Windows path are functional cross-platform (the latter is Linux-only too, actually — see AGENT_SPEC.md). The new `rotate_logs` action and drift detection are Linux-only.
- Port scan detection (also part of Phase 1 MVP's original Threat Detection promise) is still not implemented — it needs connection-tracking (not just process listing), a heavier lift than crypto-miner/drift detection; not attempted in Batch A, queued for a later batch if pursued.
- No hosted agent binary release/download flow — `install.sh` expects an operator to have already built/placed the `vigilon-agent` binary (via `go build` or a CI artifact) at `/usr/local/bin/vigilon-agent`.
- Compliance v1 covers only the 6 Vigilon-native scanner checks mapped to their nearest CIS control — not a full CIS Linux Benchmark assessment, and no other framework (ISO27001/SOC2/HIPAA/PCI-DSS) is mapped. Building real support for those frameworks needs dozens more real scanner checks, not a relabeling of the existing 6 — deliberately not attempted yet, see `ROADMAP.md`.
- The Linux-specific scanner checks (root login, world-writable scan, firewalld, rotate_logs) and RHEL-family support are verified by `go build`/`go vet` and code review only — not actually executed on a Linux or RHEL-family host in any session so far.
- Teams/Telegram/SMS channels are code-complete and verified to dispatch/fail-gracefully correctly, but none have been proven against a real Teams webhook, Telegram bot, or Twilio account (no credentials available this session).
- WhatsApp alert channel, Network Monitoring (geo map), Domain Monitoring (DNS/WHOIS/SPF/DKIM/DMARC), Windows/macOS agent parity, Kubernetes/Docker deep monitoring, and AWS/Azure/GCP cloud integrations + Cost Optimization remain unimplemented — see `ROADMAP.md` for why each is deferred.
- `CostRecommendation` and `ServerInventory` (beyond an empty read endpoint) remain schema-only — no code writes to them yet. `RemediationPolicy` is now real (Phase 2 Batch 1).
- The `block_ip` remediation action (agent-side) still only knows how to use `ufw` — it will fail on RHEL-family hosts using `firewalld`, even though the *scanner* now detects `firewalld`. Not fixed this batch; noted here rather than silently left inconsistent.
- No automated tests anywhere in the repo (backend, frontend, or agent) — verification so far is manual (build/typecheck/browser click-through/direct API calls).
- No tracked Prisma migration history (`prisma/migrations/` doesn't exist) — schema changes go through `prisma db push`, which is fine for the current single-environment dev setup but will need `prisma migrate dev` once there's a real production database to protect.

## Ground Rule Going Forward

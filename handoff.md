## What Is Actually Working (verified 2026-07-24, Phase 3 Batch F)
- **Agent (Go)**: `go build ./...` + `go vet ./...` pass. New `agent/portscan.go` closes the last piece of Phase 1 MVP's original Threat Detection promise: polls `gopsutil/v3/net.Connections("tcp")` (cross-platform) and flags a remote IP that connects to 5+ of this host's actual listening services within a 5-minute window as a `port_scan` `ThreatEvent`, with a 15-min per-IP cooldown. **10 new tests** (`agent/portscan_test.go`, 38 agent tests total now) cover the window/threshold/cooldown logic and, separately, the connection-filtering logic.
- **A real bug was found and fixed by the verification process itself, not just written and assumed correct**: the first version counted any distinct local port touched by a remote IP, including this host's own *outbound* connections â€” every outbound connection gets a fresh random ephemeral local port, so normal outbound traffic looked identical to "many distinct ports touched." Running the real compiled agent on this dev machine immediately produced false-positive `port_scan` events against `127.0.0.1`, `::1`, and a real remote IP this machine was making outbound calls to. Fixed by only counting `ESTABLISHED` connections whose local port is one this host is confirmed to be `LISTEN`ing on (i.e. actually offering a service), and excluding loopback peers entirely â€” verified both by 4 new unit tests encoding this exact scenario and by rerunning the real agent, which then completed a full cycle with zero false positives.
- **Honest scope, stated in the code comments**: this only sees currently-established sockets, not raw SYN/RST packets, and only counts hits against ports this host is actually listening on. It will catch "connect scans" (anything completing a TCP handshake, e.g. a basic `nmap -sT`) but misses stealth SYN scans and probes against closed/non-listening ports, which never become a visible connection at all.

## Known Gaps (explicit, not silently missing)
- Windows agent: security scanning and remediation are real no-ops (log a placeholder message, don't fake findings); only the core telemetry/WS loop, crypto-miner detection, port scan detection, config-sync, and Application Monitoring's Windows path are functional cross-platform (the latter is Linux-only too, actually â€” see AGENT_SPEC.md). `rotate_logs`, drift/FIM, and patch detection remain Linux-only.
- Port scan detection only catches completed-handshake "connect scans" against this host's own listening services â€” it cannot see stealth SYN scans or probes against closed ports (see Batch F above for why).
- No hosted agent binary release/download flow â€” `install.sh` expects an operator to have already built/placed the `vigilon-agent` binary (via `go build` or a CI artifact) at `/usr/local/bin/vigilon-agent`.

Building real support for those frameworks needs dozens more real scanner checks, not a relabeling of the existing 6 â€” deliberately not attempted yet, see `ROADMAP.md`.

- Teams/Telegram/SMS channels are code-complete and verified to dispatch/fail-gracefully correctly, but none have been proven against a real Teams webhook, Telegram bot, or Twilio account (no credentials available this session).
- WhatsApp alert channel, Network Monitoring (geo map), Domain Monitoring (DNS/WHOIS/SPF/DKIM/DMARC), Windows/macOS agent parity, Kubernetes/Docker deep monitoring, and AWS/Azure/GCP cloud integrations + Cost Optimization remain unimplemented â€” see `ROADMAP.md` for why each is deferred.

- Automated tests exist for the Secrets Scanner, drift/FIM's pure diff logic, the patch-check output parsers, the block_ip command builder, the remote-config merge logic, and now port-scan window/filtering logic (38 tests total as of Phase 3 Batch F, all in `agent/`) â€” everything else in the agent (most OS-interaction code, the WebSocket client, the core scan/remediation loop) plus all of backend and frontend still has zero automated test coverage; verification there remains manual (build/typecheck/browser click-through/direct API calls).

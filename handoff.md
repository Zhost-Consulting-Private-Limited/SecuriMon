## What Is Actually Working (verified 2026-07-24, Phase 3 Batch F)
- **Agent (Go)**: `go build ./...` + `go vet ./...` pass. New `agent/portscan.go` closes the last piece of Phase 1 MVP's original Threat Detection promise: polls `gopsutil/v3/net.Connections("tcp")` (cross-platform) and flags a remote IP that connects to 5+ of this host's actual listening services within a 5-minute window as a `port_scan` `ThreatEvent`, with a 15-min per-IP cooldown. **10 new tests** (`agent/portscan_test.go`, 38 agent tests total now) cover the window/threshold/cooldown logic and, separately, the connection-filtering logic.
- **A real bug was found and fixed by the verification process itself, not just written and assumed correct**: the first version counted any distinct local port touched by a remote IP, including this host's own *outbound* connections — every outbound connection gets a fresh random ephemeral local port, so normal outbound traffic looked identical to "many distinct ports touched." Running the real compiled agent on this dev machine immediately produced false-positive `port_scan` events against `127.0.0.1`, `::1`, and a real remote IP this machine was making outbound calls to. Fixed by only counting `ESTABLISHED` connections whose local port is one this host is confirmed to be `LISTEN`ing on (i.e. actually offering a service), and excluding loopback peers entirely — verified both by 4 new unit tests encoding this exact scenario and by rerunning the real agent, which then completed a full cycle with zero false positives.
- **Honest scope, stated in the code comments**: this only sees currently-established sockets, not raw SYN/RST packets, and only counts hits against ports this host is actually listening on. It will catch "connect scans" (anything completing a TCP handshake, e.g. a basic `nmap -sT`) but misses stealth SYN scans and probes against closed/non-listening ports, which never become a visible connection at all.

## What Is Actually Working (verified 2026-07-24, Phase 3 Batch H — Production Security Hardening)

This batch was triggered directly by the user asking for production-readiness and "best security standards" rather than a new feature. Audited the backend (not the whole stack) for concrete, exploitable gaps and fixed the ones that mattered without risky rewrites:

- **Critical fix — hardcoded JWT secret fallback removed.** Both `routes/auth.ts` and `middlewares/auth.ts` used to do `process.env.JWT_SECRET || 'vigilon-super-secret-key-change-in-prod'` — that literal string is public (it's in the GitHub repo), so any deployment that forgot to set `JWT_SECRET` would silently run with a secret an attacker could read straight from source and forge valid tokens for any tenant/role with. New shared `backend/src/utils/jwt.ts` throws at module load (i.e. at process startup, before the server binds) if `JWT_SECRET` is unset, empty, or equals that old literal — **verified directly**: ran the module standalone via `tsx` with `JWT_SECRET` unset (throws), with it set to the old literal (throws), and with a real random value (loads fine, confirmed via the actual dev server booting and serving `/health`). `jwt.sign`/`jwt.verify` now also pin `algorithm`/`algorithms: ['HS256']` explicitly (defense-in-depth against algorithm-confusion, even though this app has only ever used HS256).
- **Rate limiting added** (`express-rate-limit`, new dependency): a strict limiter (20 req/15min per IP) on `/v1/auth/login` and `/v1/auth/register` against credential stuffing/brute force and registration spam, plus a deliberately generous global safety-net limiter (2000 req/15min per IP) on everything else so it can't throttle real multi-server agent fleets on their normal polling cadence. **Verified end-to-end against the real running dev server**: 25 rapid bad-credential login attempts — the first ones returned 401, then 429 kicked in partway through (rate-limit window shared with earlier verification requests in the same session, which is expected/correct behavior); `/health` continued returning 200 throughout, confirming the strict limiter is scoped to the auth routes only and doesn't leak into the rest of the API.
- **`helmet` added** (new dependency) for standard secure headers (HSTS, `X-Content-Type-Options`, `X-Frame-Options`, etc.), with `crossOriginResourcePolicy` explicitly relaxed to `cross-origin` since this is a pure JSON API deliberately consumed by a separate-origin frontend (the default `same-origin` CORP would otherwise have broken the frontend's own fetch calls — caught and fixed during verification, not left as a regression). **Verified via curl against the real dev server**: confirmed `Strict-Transport-Security`, `X-Content-Type-Options: nosniff`, `X-Frame-Options: SAMEORIGIN`, and `Cross-Origin-Resource-Policy: cross-origin` all present on real responses.
- **CORS restricted to an allowlist**: `app.use(cors())` (reflected any origin) replaced with an explicit allowlist from a new `CORS_ORIGIN` env var (comma-separated), defaulting to `http://localhost:3000` so local dev needs zero extra config. **Verified via curl against the real dev server**: a request with `Origin: http://localhost:3000` gets `Access-Control-Allow-Origin` echoed back; a request with `Origin: https://evil.example` does not (no `Access-Control-Allow-Origin` header at all in the response).
- **Minimal password policy**: `/register` now rejects passwords under 8 characters with a clear 400 (previously accepted any non-empty string, including a single character). **Verified against the real dev server**: a 5-character password was rejected with `400 {"error":"Password must be at least 8 characters"}`, then registration/login/an authenticated request to `GET /v1/servers` with the issued token all succeeded with a valid password — confirming zero regression to the existing auth flow.
- `npm audit --omit=dev` re-run after adding both new dependencies: **0 vulnerabilities**, both before and after.
- Reviewed and deliberately **not** changed this batch: agent API-key auth (`routes/agent.ts`) already hashes keys with SHA-256 and looks them up via an indexed DB query — no naive string comparison, no realistic timing-attack surface, no change needed. Frontend `npm audit` flags `postcss`/`sharp` only as transitive dependencies of Next.js's own bundled build tooling; the suggested `npm audit fix --force` would downgrade `next` from v16 to v9.3.3, which is a nonsensical breaking change for a false-lead advisory in build-time-only code — reviewed and accepted, not "fixed" by a destructive downgrade.
- **Explicitly deferred, not silently dropped**: JWT is still stored in the frontend's `localStorage` rather than an `httpOnly` cookie — a real hardening opportunity (an XSS bug elsewhere in the app could exfiltrate the token), but moving it is a bigger architectural change to the auth flow that risks breaking the working login path this batch was trying to *not* break. Left as a known future item rather than attempted under this batch's "no error or glitch" constraint.

## Known Gaps (explicit, not silently missing)
- JWT tokens are stored in the frontend's `localStorage`, not an `httpOnly` cookie — a real (if standard-for-this-app-category) XSS-token-theft exposure; moving to cookie-based storage would need a broader auth-flow change and is deliberately deferred (see Batch H above) rather than attempted opportunistically.
- Windows agent: security scanning and remediation are real no-ops (log a placeholder message, don't fake findings); only the core telemetry/WS loop, crypto-miner detection, port scan detection, config-sync, generic log tailing (Batch K), and Application Monitoring's Windows path are functional cross-platform (the latter is Linux-only too, actually — see AGENT_SPEC.md). `rotate_logs`, drift/FIM, and patch detection remain Linux-only.
- Port scan detection only catches completed-handshake "connect scans" against this host's own listening services — it cannot see stealth SYN scans or probes against closed ports (see Batch F above for why).
- No hosted agent binary release/download flow — `install.sh` expects an operator to have already built/placed the `vigilon-agent` binary (via `go build` or a CI artifact) at `/usr/local/bin/vigilon-agent`.
- Compliance v1 covers only the 6 Vigilon-native scanner checks mapped to their nearest CIS control — not a full CIS Linux Benchmark assessment, and no other framework (ISO27001/SOC2/HIPAA/PCI-DSS) is mapped. Building real support for those frameworks needs dozens more real scanner checks, not a relabeling of the existing 6 — deliberately not attempted yet, see `ROADMAP.md`.
- The Linux-specific scanner/remediation checks (root login, world-writable scan, firewalld detection *and* the firewalld `block_ip` action, rotate_logs, drift/FIM, patch detection) and RHEL-family support generally are verified by `go build`/`go vet`/`go test` and code review only — not actually executed on a Linux or RHEL-family host in any session so far. This is the single biggest recurring verification gap across every batch tonight; a real Linux (ideally RHEL-family) test host would close most of it at once.
- Teams/Telegram/SMS channels are code-complete and verified to dispatch/fail-gracefully correctly, but none have been proven against a real Teams webhook, Telegram bot, or Twilio account (no credentials available this session).
- WhatsApp alert channel, Network Monitoring (geo map), Domain Monitoring (DNS/WHOIS/SPF/DKIM/DMARC), Windows/macOS agent parity, Kubernetes/Docker deep monitoring, and AWS/Azure/GCP cloud integrations + Cost Optimization remain unimplemented — see `ROADMAP.md` for why each is deferred.
- `CostRecommendation` and `ServerInventory` (beyond an empty read endpoint) remain schema-only — no code writes to them yet.
- Automated tests exist for the Secrets Scanner, drift/FIM's pure diff logic, the patch-check output parsers, the block_ip command builder, the remote-config merge logic, and port-scan window/filtering logic (38 tests total as of Phase 3 Batch F, all in `agent/`), plus (as of Phase 3 Batch I) 16 backend tests covering JWT signing/verification, the `authenticate` middleware, register/login flows, and cross-tenant server isolation (`backend/tests/`) — everything else in the agent (most OS-interaction code, the WebSocket client, the core scan/remediation loop), most of the backend (alerts, billing, remediation, AI routes, WebSocket agent comms, MSP tenant management, compliance reports), and all of the frontend still has zero automated test coverage; verification there remains manual (build/typecheck/browser click-through/direct API calls).
